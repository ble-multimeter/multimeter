# UNI-T UT219P — `ut219p`

> **State:** `untested` (ported, not bench-tested; confidence medium-low). **Driver:** `packages/protocol/src/drivers/ut219p.ts`. **Source:** ported from the UNI-T *Smart Measure* Android app (`com.uni_t.multimeter.ut219p.manager.UT219pManager` — `requestLiveDate`/`onDataReceive`/`analyseCMD0`/`setStatusFlag`); the *Bluetooth-DMM-For-Windows* note in the template does **not** apply to this driver.

The `ut219p` driver decodes the UNI-T **UT219P AC power / clamp meter**. Unlike the free-running UT60BT streamer, this model is **polled**: the host must issue a live-data request and the meter answers one frame per poll. It rides the shared ISSC "Transparent UART" service (`49535343-…`), so the registry routes it by BLE-name prefix to keep it from colliding with the `uni-t` driver. The on-wire envelope is an `AB CD` frame with a **big-endian length** and a **little-endian 16-bit checksum** stored after the length-counted region — distinct from the `uni-t` family's 19-byte big-endian-checksum frame. Everything below is derived from `ut219p.ts`; command bytes, offsets and checksums are quoted directly from the code.

## (overview)

AC power clamp meter, **polled** (does not stream). The handshake mirrors `UT219pManager`: request device-info (CMDID `0x17`), then a battery probe (CMDID `0x20`; the app refuses to poll until `deviceBattery>0`), then start polling standard live-data (CMDID `0x05`, type 0) on a host-driven cadence (~300 ms in the app). The meter answers each poll with one standard-measurement frame carrying IEEE-754 floats. The driver decodes only the **standard** measurement (CMDID `0x05`, cmdCode 0); waveform/harmonic table frames, device-info and battery replies decode to a blank reading so the chart skips them (`ut219p.ts:248-250`).

The standard-measurement frame layout, the per-value overload field, the status-bit map and all command-frame checksums are verified against the decompiled source. The `daoPos → parameter-set` dispatch is **not** decompilable (`UTDeviceBean.setValue`), so only the **ACV** set (default) and an **inferred ACA** set (`daoPos == 2`) are wired; 3P3W / 3P4W / Wh / phase-detect / power (kW/kVAr/kVA) sets are **not** implemented.

Shared ISSC "Transparent UART" service (`UT219pManager.java:57-59`, `ut219p.ts:19-21, 400`):

**Routing.** The `uni-t` driver advertises the same ISSC service, so the registry cannot disambiguate by service UUID; it routes by the `UT219P` / `UT-219P` **name prefix** instead (`ut219p.ts:17-18, 402-404`). The driver also supplies a `sniff()` that returns `checksumOk(bytes)` (`ut219p.ts:431-433`) — a UT219P reply is an `AB CD` envelope whose big-endian length and trailing little-endian checksum line up, distinct from the `uni-t` 19-byte big-endian-checksum frame.

## Handshake / session start

The meter never free-runs. `handshake(io)` (`ut219p.ts:408-421`) runs a three-step sequence and then nudges the live poll until the first measurement arrives:

1. **Device-info** — write `DEVINFO_REQ` (CMDID `0x17`), wait up to 1500 ms for any `control` frame.
2. **Battery probe** — write `BATTERY_REQ` (CMDID `0x20`), wait up to 1000 ms for any `control` frame. (In the app this gates polling: it refuses to poll until `deviceBattery>0`.)
3. **Live poll** — up to 5 attempts: write `LIVE_STD` (CMDID `0x05`, type 0) and wait up to 700 ms for a `measurement` frame; return on the first one. If none arrives after 5 attempts, throw `UT219P did not answer live-data poll after handshake`.

```ts
async handshake(io) {
  await io.write(DEVINFO_REQ);
  await io.waitForFrame(k => k === 'control', 1500);
  await io.write(BATTERY_REQ);
  await io.waitForFrame(k => k === 'control', 1000);
  for (let attempt = 0; attempt < 5; attempt++) {
    await io.write(LIVE_STD);
    if (await io.waitForFrame(k => k === 'measurement', 700)) return;
  }
  throw new Error('UT219P did not answer live-data poll after handshake');
}
```

After the handshake the host keeps polling on its own cadence (the session's keep-alive timer re-issues `LIVE_STD`); `onRequest()` is a no-op because the meter never asks the host to re-send (`ut219p.ts:424-427`).

### Command frames (exact bytes, checksums verified)

All literal request frames are quoted from the source (`ut219p.ts:35-40`). Small command frames carry `LEN=0x0004` and store their 16-bit little-endian checksum immediately after the length-counted region (checksum over `bytes[2 .. 2+LEN)`, stored at `[2+LEN]`/`[2+LEN+1]`).

| Frame | Bytes (hex) | CMDID | Notes |
| --- | --- | --- | --- |
| `DEVINFO_REQ` | `AB CD 00 04 17 00 1B 00` | `0x17` | device-info request; chk = `0x04+0x17 = 0x1B` |
| `BATTERY_REQ` | `AB CD 00 05 20 E8 03 13 01` | `0x20` | **special**: `LEN=0x0005`; trailing bytes are a fixed probe payload, **not** a checksum (`UT219pManager:667` sends verbatim) |
| `LIVE_STD` | `AB CD 00 04 05 00 09 00` | `0x05` | standard live-data poll (`liveDataReq(0)`); chk = `0x04+0x05+0x00 = 0x09` |
| `LOCK` | `AB CD 00 04 19 5A 77 00` | `0x19` | freeze device readout (for waveform capture); chk = `0x04+0x19+0x5A = 0x77` |
| `UNLOCK` | `AB CD 00 04 19 A5 C2 00` | `0x19` | unfreeze; chk = `0x04+0x19+0xA5 = 0xC2` |
| `OTA` | `AB CD 00 04 90 00 94 00` | `0x90` | firmware update; chk = `0x04+0x90 = 0x94` |

These are also re-exported as `COMMANDS = { DEVINFO_REQ, BATTERY_REQ, LIVE_STD, LOCK, UNLOCK, OTA }` for callers that drive lock/waveform/OTA directly (`ut219p.ts:449-456`).

**Live-data poll builder** (`liveDataReq(type)`, `ut219p.ts:46-58`). The standard poll is `liveDataReq(0)`; other screen types are `7`=wave-V, `8`=wave-A, `3/4/5/6`=harmonic variants (`ut219p.ts:42-43`). The frame is:

```
AB CD 00 04 05 <type> <(type+9) & 0xFF> <((type+9) >> 8) & 0xFF>
```

The trailing two bytes are the little-endian checksum `= sum(bytes[2..6)) = 0x04 + 0x05 + type = type + 9`.

## Frame format

### Reply envelope

Parser convention from `UT219pManager.onDataReceive` (`ut219p.ts:60-65`):

| Offset | Field |
| --- | --- |
| `[0]` | SOF0 = `0xAB` |
| `[1]` | SOF1 = `0xCD` |
| `[2..3]` | `LEN` — **big-endian** (`payloadLen = (b[2]<<8) | b[3]`, `ut219p.ts:63-65`) |
| `[4]` | CMDID |
| `[5 …]` | body |
| `[i+4]`,`[i+5]` | 16-bit **little-endian** checksum, where `i = LEN` |

Total reply length = `i + 6`. The checksum covers `bytes[2 .. i+4)`.

### Checksum (`checksumOk`, `ut219p.ts:69-78`)

Returns `false` on any shape problem (so the framer/decoder degrades gracefully rather than throwing):

```ts
if (frame.length < 6) return false;
if (frame[0] !== SOF0 || frame[1] !== SOF1) return false;
const i = payloadLen(frame);
if (i > frame.length - 6) return false;
let sum = 0;
for (let k = 2; k < i + 4; k++) sum += frame[k]!;
sum &= 0xffff;
return (sum & 0xff) === frame[i + 4] && ((sum >> 8) & 0xff) === frame[i + 5];
```

### Framer (`Ut219pFramer`, `ut219p.ts:349-393`)

Buffers chunks and slices the `AB CD` envelope by the big-endian `LEN` (total = `LEN+6`), validating the trailing little-endian checksum; tolerates split/coalesced notifications like the `uni-t` `FrameParser`.

- `sync()` (`ut219p.ts:380-392`) discards leading bytes until the buffer starts with `0xAB`, and if a second byte is present discards again unless it is `0xCD`.
- Needs ≥4 bytes for `AB CD LEN_h LEN_l`; computes `total = LEN + 6`.
- Rejects a bogus length (`LEN < 1` or `total > 2048`) by shifting one byte and resyncing (`ut219p.ts:360-363`).
- Waits if the frame is split across notifications (`buf.length < total`).
- On checksum failure, shifts one byte and resyncs past the false boundary (`ut219p.ts:366-369`).
- Valid frames are emitted with a `kind` from `classify()`.

### Frame classification (`classify`, `ut219p.ts:341-345`)

```ts
if (frame.length < 6) return 'control';
if (frame[4] === CMD_LIVE && frame[5] === 0) return 'measurement';
return 'control';
```

Only **CMDID `0x05` cmdCode `0`** (standard live-data) frames are `measurement`; device-info, battery and everything else are `control` (they gate the handshake).

### Standard-measurement layout (`analyseCMD0`)

Decoded fields and their byte offsets within the reply frame (`ut219p.ts:252-285`):

| Offset | Field | Meaning |
| --- | --- | --- |
| `[4]` | CMDID | must be `0x05` (`CMD_LIVE`) |
| `[5]` | cmdCode | must be `0` (standard); 7/8 = waveform, 3-6 = harmonic |
| `[6]` | `daoPos` | parameter-set selector (dispatch inferred — see Decode) |
| `[7]` | statusByte1 | flags (see below) |
| `[8]` | statusByte2 | flags (see below) |
| `[11]` hi-nibble | `idx` | first nibble index code → primary line in the active set (`(b[11] & 0xf0) >> 4`) |
| `[13..16]` | overload field | 16 two-bit per-value codes, MSB-first (see below) |
| `[19..22]` | first float | primary line value, 32-bit LE IEEE-754 |

The decoder requires `bytes.length >= 23` to reach the first float (`ut219p.ts:253`). The float count is `(LEN-15)/4`, with the first float at byte 19 (`ut219p.ts:284`); only this primary float is decoded into a `Reading`.

**Float decode** (`floatLE`, `ut219p.ts:82-87`). 32-bit LE IEEE-754: `buf[n]` is the LSB. Mirrors the source's `Float.intBitsToFloat(getIntValue(buf[n+3],buf[n+2],buf[n+1],buf[n]))`:

```ts
const bits = (frame[n+3] << 24) | (frame[n+2] << 16) | (frame[n+1] << 8) | frame[n];
dv.setInt32(0, bits, false); return dv.getFloat32(0, false);
```

**Per-value overload codes** (`overloadCodes`, `ut219p.ts:89-99`). Bytes `[13..16]` are read as a 32-bit value and unpacked into 16 two-bit fields, **MSB-first**; the first field (`codes[0]`) applies to the primary line:

| Code | Meaning | Display |
| --- | --- | --- |
| `0` | normal | numeric value |
| `1` | OL (overload) | `OL`, `overload = true` |
| `2` | Err | `Err` |
| `3` | `--` (negative overload) | `--`, `overload = true` |

(`OVERLOAD_TEXT = {1:'OL', 2:'Err', 3:'--'}`, `ut219p.ts:230`; `overload = code0 === 1 || code0 === 3`, `ut219p.ts:270`.)

**Status-bit map** (`parseFlags`, statusByte1 `b[7]` / statusByte2 `b[8]`, `setStatusFlag`, `ut219p.ts:101-125`):

| Flag | Source | Decode |
| --- | --- | --- |
| max / min / avg | `b1 & 0xC0 >> 6` | `0` none, `1` MAX, `2` MIN, `3` AVG → `max = (==1)`, `min = (==2)` |
| `hold` | `b1 & 0x10` | set = hold |
| `rel` | `b2 & 0x08` | set = rel |
| `auto` | `b1 & 0x02` | bit clear = auto; **set = manual range** (`auto = (b1 & 0x02) === 0`) |
| `rangeValue` | `b1 & 0x0C >> 2` | range index (drives decimals) |
| `hvWarning` / `hv` | `b2 & 0x04` | high-voltage warning |
| `thd` | `b2 & 0x10` | THD active (internal; not surfaced on `Reading.flags`) |
| `lowBattery` | — | always `false` — battery arrives via the separate CMDID `0x20` reply, not the status byte |
| `peakMax` / `peakMin` | — | always `false` |

## Decode

`decodeUt219p(bytes, ts)` (`ut219p.ts:242-324`) is pure and never throws; it degrades gracefully. Guards (each returns a `blank` reading, shape at `ut219p.ts:204-228`):

- `bytes.length < 6` → blank (`ut219p.ts:243`).
- not `AB CD` → blank (`ut219p.ts:244`).
- bad checksum → blank (`ut219p.ts:245`).
- CMDID `!= 0x05` → blank (devinfo / battery carry no reading) (`ut219p.ts:247-248`).
- cmdCode `!= 0` → blank (waveform/harmonic frames are not chart readings) (`ut219p.ts:249-250`).
- `bytes.length < 23` → blank (need at least one float) (`ut219p.ts:253`).

### Parameter-set dispatch (the implemented gap)

Each parameter set is a 1-based `{index → {title, unit}}` table keyed by the nibble index code; the precise `daoPos → set` dispatch lives in the **non-decompiled** `UTDeviceBean.setValue`, so it is **inferred** (`ut219p.ts:127-131, 326-337`):

```ts
function chooseSet(daoPos) {
  switch (daoPos) {
    case 2:  return ACA_SET;   // inferred
    default: return ACV_SET;   // confident default
  }
}
```

Only two sets are tabled:

**`ACV_SET`** (`setACVParam`, index 1..9, `ut219p.ts:138-148`):

| idx | title | unit | | idx | title | unit |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | `ACV` | `V` | | 6 | `V(MIN)` | `V` |
| 2 | `V(PEAK)` | `V` | | 7 | `TIME(MIN)` | `` |
| 3 | `Hz(V)` | `Hz` | | 8 | `V(AVG)` | `V` |
| 4 | `V(MAX)` | `V` | | 9 | `TIME` | `` |
| 5 | `TIME(MAX)` | `` | | | | |

**`ACA_SET`** (`setACAParam`, index 1..9, `ut219p.ts:151-161`):

| idx | title | unit | | idx | title | unit |
| --- | --- | --- | --- | --- | --- | --- |
| 1 | `ACA` | `A` | | 6 | `A(MIN)` | `A` |
| 2 | `A(PEAK)` | `A` | | 7 | `TIME(MIN)` | `` |
| 3 | `Hz(A)` | `Hz` | | 8 | `A(AVG)` | `A` |

The selected line is `set[idx] ?? set[1] ?? { title: '#${daoPos}.${idx}', unit: '' }` (`ut219p.ts:267`).

**Not implemented:** 3P3W / 3P4W / Wh / phase-detect / power (kW/kVAr/kVA) parameter sets are absent. The `functionFor` / `decimalsFor` helpers already anticipate `kW`/`kVAr`/`kVA`/`°`/`%`/`Wh` units (`ut219p.ts:165-201`), but no table feeds them because the `daoPos`→set dispatch for those modes is unresolved. The nibble→line mapping for multi-phase frames is also unknown. The driver's own comment is explicit: "the daoPos→parameter-set dispatch (UTDeviceBean.setValue) was not decompilable and is inferred" (`ut219p.ts:11-12`).

### Value, decimals, units

- **Decimals** (`decimalsFor`, `ut219p.ts:165-174`): `ACV`/`V` → 1 dp; `A` (and `kW`/`kVAr`/`kVA`) → by `rangeValue` via `[2,1,0,0]`; `PF` → 3 dp; `Hz`/`°`/`%` → 1 dp; default 1 dp.
- **Overload path** (`code0 !== 0`, `ut219p.ts:287-302`): returns `displayText = OVERLOAD_TEXT[code0]`, `displayValue = null`, blanked unit, `acdc = 'AC'` for V/A lines.
- **Numeric path** (`ut219p.ts:304-323`): `displayText = value.toFixed(dp)`; `displayValue = Number(displayText)`; `displayUnit = line.unit`; `{ base, exp } = unitInfo(displayUnit)` (`types.ts:168-174`); `baseValue = displayValue * 10**exp`.
- **`acdc`**: `'AC'` when the line unit is `V` or `A`, else `''` (`ut219p.ts:298, 320`) — this is an AC meter, so all V/A readings are AC.
- **`bargraph`**: always `0` (`ut219p.ts:300, 322`).

### Function key (`functionFor`, `ut219p.ts:178-201`)

Maps the line's unit to a stable function key: `V → ACV`, `A → ACA`, `Hz → Hz`, `kW/kVAr/kVA → kW/kVAr/kVA`, `° → angle`, `% → THD`, `Wh → Wh`, else `title || '?'`. Feeds `quantityKey` (`types.ts:49-51`) as `function|acdc` for chart-segment splitting.

## Controls

UT219P has **no** `AB CD 03` soft-button scheme; the app's HOLD is local-only (no BLE frame). The driver maps the generic control slots to the closest real device action (`ut219p.ts:435-444`):

| Control | Command | Frame |
| --- | --- | --- |
| `hold` | `LOCK` | `AB CD 00 04 19 5A 77 00` — freeze the device readout (for waveform capture) |
| `rel` | `UNLOCK` | `AB CD 00 04 19 A5 C2 00` — unfreeze |

`OTA` (`AB CD 00 04 90 00 94 00`) is exported via `COMMANDS` for callers that drive firmware update directly, but is not wired into the generic control set.

## Verification

`verification: 'ported-unverified'` (`ut219p.ts:398`). **Confidence: medium-low.** Ported from `UT219pManager` (`analyseCMD0` / `setStatusFlag` / `requestLiveDate`) and **not** bench-tested on physical hardware.

Verified against the decompiled source:
- The standard-measurement frame layout, the per-value overload field, the status-bit map and all command-frame checksums.

Inferred / unimplemented (the gap):
- `daoPos → parameter-set` dispatch (`UTDeviceBean.setValue` not decompilable). Only ACV (default) and the inferred ACA set (`daoPos == 2`) are wired; 3P3W / 3P4W / Wh / phase-detect / power (kW/kVAr/kVA) sets are **not** implemented.
- The nibble→line mapping for multi-phase frames is unknown.
- `lowBattery`, `peakMax`, `peakMin` are not in the standard frame and are hard-coded `false` (battery arrives via the separate CMDID `0x20` reply).
- The framer's split/coalesced-notification handling mirrors the `uni-t` parser defensively; multi-frame resync is untested against real traffic.

**Needs:** per-mode hardware capture (ACV, ACA, 3P3W, 3P4W, Wh, phase-detect) to confirm the `daoPos` values and the nibble→line mapping, and to validate float offsets/decimals on a live device.

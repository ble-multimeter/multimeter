# UNI-T UT181A — `ut181a`

> **State:** `untested` (ported, not bench-tested; RX vectors synthesized). **Driver:** `packages/protocol/src/drivers/ut181a.ts`. **Source:** ported from `webspiderteam/Bluetooth-DMM-For-Windows`.

The `ut181a` driver decodes the UNI-T **UT181A** datalogging true-RMS bench/handheld multimeter. It physically shares the ISSC "Transparent UART" GATT service with the other UNI-T BLE meters, but speaks its **own** `AB CD` envelope protocol, distinct from the generic uni-t 19-byte frame and structurally identical to the UT171's little-endian length-prefixed container. Because the service alone is not distinguishing, discovery is **routed by advertised name** (the name contains `UT181`) and disambiguated by a frame `sniff()`.

The meter does not free-stream on subscribe: the client must send a START command (`createCmd(5, {0x01})`) to begin the live data transfer, after which the meter pushes opcode-`0x02` LIVEDATA frames continuously. Soft-button controls (HOLD / MAX-MIN / RANGE / SELECT) map to additional `AB CD` command frames. Everything below is derived from `ut181a.ts`; envelope, checksum, command and decode details are quoted directly from the code.

## Models

| Model | BLE name prefixes |
| --- | --- |
| UT181A | `UT181A`, `UT181` |

`namePrefixes` are `['UT181A', 'UT181']` (`ut181a.ts:319`). `match()` returns true iff the advertised name *contains* `"UT181"` (`ut181a.ts:322`) — it deliberately does **not** key on the shared ISSC service, so the generic uni-t driver can still claim other UNI-T meters on that same service.

## Transport (GATT)

The ISSC "Transparent UART" profile, shared with the other UNI-T BLE meters (`ut181a.ts:307-320`):

| Role | UUID |
| --- | --- |
| Service | `49535343-fe7d-4ae5-8fa9-9fafd205e455` |
| Notify | `49535343-1e4d-4bd9-ba61-23c647249616` |
| Write (preferred) | `49535343-8841-43f4-a8d4-ecbe34729bb3` |
| Write (fallback) | `49535343-6daa-4d02-abf6-19569aca69fe` |

`gatt.write` lists both write characteristics as candidates (`WriteUUID2` first, preferred when present; first match wins) to tolerate firmware reshuffles (`ut181a.ts:320`).

Because multiple drivers share this service, identification is **name-routed**: `match()` keys on the `UT181` name substring, and `sniff()` (`looksLikeUt181aFrame`) confirms a raw notification is an `AB CD`-envelope live frame before the session commits this decoder (`ut181a.ts:322, 353, 297-305`).

## Handshake / session start

### Envelope builder (`createCmd`, `ut181a.ts:35-49`)

Every TX command is built by `createCmd(opcode, body)`:

```
AB CD <lenLo lenHi> <opcode> <body...> <chkLo chkHi>
```

* `len16 = body.length + 3` (= `total_len − 4`); stored **little-endian** as `lenLo, lenHi` (`ut181a.ts:36, 39-40`).
* `chk16 = (Σ bytes[2 .. end-of-body]) & 0xFFFF` — the sum runs from the length field (index 2) through the last body byte, i.e. it covers `lenLo, lenHi, opcode, body...` and excludes the `AB CD` header and the checksum itself; stored **little-endian** as `chkLo, chkHi` (`ut181a.ts:45-48`).

### Sequence (`handshake`, `ut181a.ts:329-335`)

1. Write `CMD_START` (`createCmd(5, {0x01})`).
2. `waitForFrame(kind === 'measurement', 1500ms)`. If a live frame lands, return.
3. Retry up to **3 attempts** total (the upstream re-issues the start command at roughly +3000 ms and +5000 ms). After 3 failed attempts the handshake returns best-effort, leaving the stream armed.

No request/response keep-alive frames are emitted by this meter — `onRequest()` is a no-op (`ut181a.ts:338-340`); the START command is the only nudge.

### TX command vectors (byte-exact)

All built via `createCmd` (`ut181a.ts:288-292`). The two short vectors marked **exact** are byte-verified against the spec; the rest follow the same envelope/checksum derivation.

| Name | `createCmd(...)` | Bytes | Note |
| --- | --- | --- | --- |
| START (live data) | `createCmd(5, [0x01])` | `AB CD 04 00 05 01 0A 00` | `CMDID_DATA_TREAFER {0x01}`, **exact** |
| HOLD | `createCmd(18, [0x5A])` | `AB CD 04 00 12 5A 70 00` | `CMDID_HOLD` magic `0x5A`, **exact** |
| MAX/MIN (enter) | `createCmd(4, [0x01])` | `AB CD 04 00 04 01 09 00` | `CMDID_MaxMin` enter |
| RANGE | `createCmd(2, [0x00])` | `AB CD 04 00 02 00 06 00` | `CMDID_Range`, idx `0` = cycle |
| SELECT | `createCmd(1, [0x11, 0x00])` | `AB CD 05 00 01 11 00 17 00` | `CMDID_CHANGE_FUNC`, low byte `0x11` = select group |

(START/HOLD/MAX-MIN/RANGE bytes are asserted byte-for-byte by the unit tests; `ut181a.test.ts:180-187, 219-221`.)

## Frame format

The RX (and TX) envelope is the little-endian length container, identical in shape to the UT171 (`ut181a.ts:10-22`):

```
[0]      0xAB                              header byte 0
[1]      0xCD                              header byte 1
[2..3]   lenLo lenHi    len16 = payloadLen + 3 = total_len − 4   (LE)
[4]      opcode
[5..]    body           (len16 − 1 bytes)
[last-1] chkLo
[last]   chkHi          chk16 = (Σ bytes[2 .. 4+payloadLen-1]) & 0xFFFF   (LE)
```

* `OPCODE_LIVE = 0x02` is the live measurement frame; the framer surfaces it as `'measurement'`, all other opcodes as `'control'` (`ut181a.ts:29, 262`).
* `total = len16 + 4` must equal `frame.length` (`ut181a.ts:55-57`).

### Checksum validation (`checksumOk`, `ut181a.ts:52-62`)

A received frame is valid iff:

* `frame.length >= 7`, and
* `frame[0] === 0xAB && frame[1] === 0xCD`, and
* `len16 + 4 === frame.length`, and
* `(Σ frame[2 .. total-3]) & 0xFFFF` matches the trailing LE pair: low byte `== frame[total-2]`, high byte `== frame[total-1]`.

The sum covers the two length bytes, the opcode and the body — excluding the header and the checksum bytes themselves.

### Live measurement frame body (opcode `0x02`)

Body offset 0 is `frame[5]` (`ut181a.ts:17-20`):

| Offset | Field |
| --- | --- |
| `frame[5]` | `flagsA` |
| `frame[6]` | `flagsB` |
| `frame[7..8]` | `measureCode` (LE u16) |
| `frame[9]` | `rangeIndex` |
| `frame[10..]` | value block (the MAIN value, see below) |

### Value block (`parseCommonData`, base index `b`)

For the MAIN value, `b = 10` (`ut181a.ts:178`):

| Offset | Field |
| --- | --- |
| `b .. b+3` | `float32` value, **wire little-endian** IEEE-754 |
| `b+4` | status/scale byte: **low nibble** = OL/status code, **high nibble** = decimal places (`dot`) |
| `b+5 .. b+12` | 8-byte ASCII unit string, NUL-terminated |

The float read (`readFloatLE`, `ut181a.ts:107-110`) is `DataView.getFloat32(0, true)`. The upstream reverses the 4 wire bytes into a big-endian int and reconstructs the float; reading the 4 bytes little-endian is equivalent.

The in-band ASCII unit (`decodeUnit`, `ut181a.ts:93-103`) reads up to 8 bytes from `b+5`, breaking on NUL, translating `0xB0 → "°"` and `0x7E → "Ω"`, then `.trim()`s. Because the **human-readable unit including the metric prefix is sent in-band**, the driver never needs the absent `measureCode → name` JSON table to render value + unit.

### MAX/MIN / COMP / record sub-frames

The driver decodes only the **MAIN** value block (`b = 10`). The upstream `parseCommonData` is a generic value-block reader (float + status/scale + 8-byte unit) reused for the additional MAX / MIN / COMP / peak and stored-record sub-frames that follow the MAIN block in some opcode-`0x02` payloads — but this port reads only the first (MAIN) block and ignores any trailing sub-frames (`ut181a.ts:17-20, 169, 178`). The framer still reassembles and length-validates the full frame regardless of how many sub-blocks it carries.

### Decimal rounding

`dot = (statusByte >> 4) & 0x0F`. With `f = 10 ** dot`, the displayed value is `Math.round(raw * f) / f` and `displayText = displayValue.toFixed(dot)` (`ut181a.ts:181, 204-206`).

## Decode

`decodeUt181a(bytes, ts)` (`ut181a.ts:168-234`) is pure and never throws — a wrong-opcode, short, or checksum-failing frame returns a `blank` reading (`ut181a.ts:65-89`) rather than raising.

### Gate (returns `blank` on any failure, `ut181a.ts:170-173`)

* `bytes.length >= 23` (the MAIN block needs `frame[5..22]`: flags + code + range + float + status + 8-byte unit),
* `bytes[0] === 0xAB && bytes[1] === 0xCD`,
* `bytes[4] === 0x02` (live opcode),
* `checksumOk(bytes)`.

### Field derivation

* **value / status:** `raw = readFloatLE(bytes, 10)`; `statusByte = bytes[14]`; `dot = statusByte>>4`; `status = statusText(statusByte & 0x0F)`.
* **status nibble** (`statusText`, `ut181a.ts:144-161`, low nibble of `b+4`):

  | Nibble | Text | Meaning |
  | --- | --- | --- |
  | 0 | (null) | normal |
  | 1 | `OL` | over-range |
  | 2 | `-OL` | negative over-range |
  | 4 | `LEAD` | lead warning |
  | 5 | `DISC` | discontinuity |
  | 6 | `Lo` | low (comparator) |
  | 7 | `Hi` | high (comparator) |

* **displayUnit:** `decodeUnit(bytes, 15)` (the 8-byte in-band string).
* **base unit + exponent:** `unitInfo(displayUnit)` (`types.ts:168-174`) splits a metric prefix off the front (`n/µ/m/k/M`) only when a base follows, giving `base` and `exp` such that `baseValue = displayValue * 10^exp`.
* **AC/DC** (`ut181a.ts:188-190`): not a dedicated live-frame flag. The decoded unit string is upper-cased; `AC` substring → `'AC'`, else `DC` substring → `'DC'`, else `''`. Exact AC/DC for ranges whose in-band unit omits the suffix would need a `measureCode → name` table, which is not available, so it falls back to `''` (DC/none).
* **function label** (`functionFor(baseUnit, acdc)`, `ut181a.ts:115-140`): derived from the **decoded base unit** (not `measureCode`, since the friendly-name table is not available), giving a range-independent key so range changes stay one chart segment while a real mode change splits it:

  | base unit | function |
  | --- | --- |
  | `V` | `ACV` / `DCV` / `V` (acdc-prefixed when known) |
  | `A` | `ACA` / `DCA` / `A` |
  | `Ω` | `OHM` |
  | `F` | `CAP` |
  | `Hz` | `Hz` |
  | `%` | `%` |
  | `s` | `s` |
  | `dBm` | `dBm` |
  | `°C` | `°C` |
  | `°F` | `°F` |
  | other | the base unit itself, or `'?'` if empty |

### Value/display resolution (`ut181a.ts:192-209`)

* `overload = status === 'OL' || status === '-OL'`.
* If `status !== null` (over-range / lead / discontinuity / comparator): `displayValue = null`, `displayText = status`, `baseValue = null`.
* Else if `raw` is not finite: `displayValue = null`, `displayText = ''`.
* Else: `displayValue = round(raw·10^dot)/10^dot`, `displayText = displayValue.toFixed(dot)`, `baseValue = displayValue · 10^exp`.

### Flags (`ut181a.ts:222-232`)

| Reading flag | Source | Note |
| --- | --- | --- |
| `max` | `flagsA & 0x20` (bit 5) | `isMaxMin`; sets both `max` and `min` |
| `min` | `flagsA & 0x20` (bit 5) | same bit (no separate min bit in this layout) |
| `hold` | `flagsA & 0x80` (bit 7) | `isHold` |
| `rel` | `flagsA & 0x10` (bit 4) | `isRel` |
| `auto` | `flagsB & 0x01` (bit 0) | `isAuto` |
| `lowBattery` | always `false` | no low-battery bit in this frame layout |
| `hvWarning` | `flagsB & 0x02` (bit 1) | `isHV` |
| `peakMax` | `flagsA & 0x40` (bit 6) | `isPeak`; sets both peak flags |
| `peakMin` | `flagsA & 0x40` (bit 6) | same bit |

`bargraph` is always `0` (this frame carries no analog bar).

### Framer (`Ut181aFramer`, `ut181a.ts:239-284`)

Buffered `AB CD` reassembly that tolerates split/coalesced notifications:

1. `sync()` drops bytes until the buffer head is `0xAB 0xCD`.
2. Needs ≥ 4 bytes to read `len16`; `total = len16 + 4`.
3. If `total < 7 || total > 512`, the head byte is a false `AB CD` in noise → shift one byte and resync.
4. If fewer than `total` bytes buffered → wait (frame split across notifications).
5. Slice `total` bytes; if `checksumOk` fails → shift one byte and resync (desync, not a real boundary).
6. On success, emit `{ kind: opcode === 0x02 ? 'measurement' : 'control', bytes }`.

`reset()` clears the buffer.

### Sniffer (`looksLikeUt181aFrame`, `ut181a.ts:297-305`)

Returns true iff `bytes.length >= 7 && bytes[0] === 0xAB && bytes[1] === 0xCD && bytes[4] === 0x02 && checksumOk(bytes)`. This distinguishes a UT181A live frame from the generic uni-t 19-byte frame (which has no `AB CD` LE-checksum of this form and places its opcode differently).

## Controls

Front-panel soft buttons (`ut181a.ts:346-351`), each a fixed `AB CD` command frame:

| `MeterControl` | Command | Bytes |
| --- | --- | --- |
| `hold` | `CMD_HOLD` | `AB CD 04 00 12 5A 70 00` |
| `maxMin` | `CMD_MAXMIN_ENTER` | `AB CD 04 00 04 01 09 00` |
| `range` | `CMD_RANGE` | `AB CD 04 00 02 00 06 00` (idx 0 cycles) |
| `select` | `CMD_SELECT` | `AB CD 05 00 01 11 00 17 00` (changes function) |

No `backlight`, `hzDuty`, `rel`, or `rangeAuto` opcode exists for this model (`ut181a.ts:344-345`). RANGE idx `0` cycles the range; SELECT changes the function/group.

## Verification

`verification: 'ported-unverified'` (`ut181a.ts:316`; `ut181a.test.ts:201-205`). **Confidence: medium.**

* **Not bench-tested** on physical hardware.
* **TX vectors are byte-exact:** START, HOLD, MAX-MIN and RANGE are asserted byte-for-byte against the spec (`ut181a.test.ts:180-187, 219-221`); `createCmd` is verified against the byte-exact start (`AB CD 04 00 05 01 0A 00`) and hold (`AB CD 04 00 12 5A 70 00`) vectors (`ut181a.ts:34`).
* **RX vectors are synthesized:** no captured demo byte arrays are available for this model, so the live-frame test vectors (DCV, OHM, °C, OL/-OL, HOLD) are constructed from the documented `parseCommonData` layout with length/checksum bytes recomputed to be byte-accurate (`ut181a.test.ts:1-38`). The decode logic, framer and sniffer pass against these synthesized frames, but the byte layout itself is unconfirmed on hardware.
* **Function-name table not available:** there is no `measureCode → friendly-name` table, so the function label and exact AC/DC are derived from the **in-band unit string** instead of the measure code. AC/DC for ranges whose in-band unit omits an `AC`/`DC` suffix degrades to `''`.

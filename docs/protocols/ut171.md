# UNI-T UT171 (A/B/C) — `ut171`

> **State:** `untested` (ported, not bench-tested). **Driver:** `packages/protocol/src/drivers/ut171.ts`. **Source:** ported from `webspiderteam/Bluetooth-DMM-For-Windows`.

The `ut171` driver speaks UNI-T's **AB-CD length-prefixed protocol** as used by the UT171-series true-RMS bench/clamp multimeters, typically reached through the **UT-D07A** IR/optical-to-BLE adapter. It rides the same ISSC "Transparent UART" GATT service as the UT60BT/UT161 family (`ut171.ts:6-7`) but carries a completely different wire format: a little-endian length container, little-endian IEEE-754 32-bit float measurement values, a little-endian additive checksum, and per-value unit codes (the older `uni-t` driver decodes a literal ASCII LCD string with a big-endian checksum over a fixed 19-byte frame). The meter does **not** free-run; the handshake turns the live stream on and the meter then pushes `CMDID=2` measurement frames continuously (`ut171.ts:16-18`).

This driver was ported from the decompiled UNI-T "Smart Measure" Android app (`manager/UT171AManager.java`, `bean/InnerRecordBean.java`, `ui/main/anjianview/Anjianview171.java`, `manager/BleManager.java`) and the upstream `webspiderteam/Bluetooth-DMM-For-Windows` project, and has **not** been tested against physical hardware, so its `Driver.verification` is `'ported-unverified'` (`ut171.ts:377`). Markers below: ⚠️ present in code but unverified against a physical unit · ❓ inferred / best-guess mapping that needs hardware confirmation.

## Models

| Model | BLE name prefix | State |
|---|---|---|
| UT171A / UT171B / UT171C | `UT171`, `UT171C` | ⚠️ `untested` — ported decoder, not hardware-verified |
| UT-D07A IR/BLE adapter | `UT-D07A` | ⚠️ `untested` — the adapter advertises its own name and is matched too |

`namePrefixes` is `['UT171', 'UT171C', 'UT-D07A']` (`ut171.ts:381`). The whole A/B/C family is handled by one decoder — there is no per-variant branch in the code. The `UT-D07A` adapter is the usual transport: it bridges the meter's optical port to BLE and advertises a `UT171`/`UT-D07A` name, so the driver claims it.

## Transport (GATT)

A Microchip/ISSC **"Transparent UART"** module: one notify characteristic (meter → app, the measurement stream) and two candidate write characteristics (app → meter, commands). The driver prefers the `8841` write-no-response characteristic when present and falls back to the `6daa` characteristic (`ut171.ts:25-28`).

| Role | Full UUID | Source |
|---|---|---|
| Service (ISSC Transparent UART) | `49535343-fe7d-4ae5-8fa9-9fafd205e455` | `ut171.ts:23` |
| Notify (meter→app) | `49535343-1e4d-4bd9-ba61-23c647249616` | `ut171.ts:24` |
| Write (app→meter, preferred) | `49535343-8841-43f4-a8d4-ecbe34729bb3` | `ut171.ts:27` |
| Write fallback | `49535343-6daa-4d02-abf6-19569aca69fe` | `ut171.ts:28` |

The `gatt` descriptor (`ut171.ts:382`) lists `write` as an ordered array `[ISSC_WRITE, ISSC_WRITE_FALLBACK]` so the transport falls back to `6daa` if `8841` is unwritable.

**Routing (name-based).** Because the ISSC service is *shared* with UT60BT/UT161/UT181, the driver does **not** match on the service UUID. Its `match` (`ut171.ts:384-385`) returns true only when the advertised device name starts with `UT171` **or** `UT-D07A`. The registry routes ISSC matches by name so this driver claims only UT171-family devices, and `sniff` (`looksLikeUt171Frame`, see below) disambiguates the raw frame format when more than one driver shares the service.

## Handshake / session start

The meter is silent until told to stream. `handshake(io)` (`ut171.ts:392-400`):

1. Write **START** (`UT171_COMMANDS.START`, cmd `10` `{0x01}`) — `CMDID_DATA_TREAFER`, turns the live stream on.
2. Write **DEVICE_INFO** (`UT171_COMMANDS.DEVICE_INFO`, cmd `22` `{0x5A}`) — `CMDID_REQUESTDEVICEINFO`.
3. **Re-nudge loop**, up to **3×**: `await io.waitForFrame(k => k === 'measurement', 1000)`; return on the first measurement frame. If none arrives within 1 s, re-send **START** and try again. This mirrors the app's re-arm for the UT-D07A adapter (the adapter sometimes needs START repeated).

After START the meter pushes `CMDID=2` LIVEDATA frames on its own — there is **no** request/response keep-alive, so `onRequest()` is a no-op (`ut171.ts:402-405`). If all 3 nudges fail the handshake simply returns (it does not throw).

The handshake/control bytes (exact on-wire hex):

| Command | Bytes (hex) | cmd | Payload | Purpose | Source |
|---|---|---|---|---|---|
| START | `AB CD 04 00 0A 01 16 00` | `0x0A` (10) | `{0x01}` | `CMDID_DATA_TREAFER` — enable live stream | `ut171.ts:344` |
| DEVICE_INFO | `AB CD 04 00 16 5A 7B 00` | `0x16` (22) | `{0x5A}` | `CMDID_REQUESTDEVICEINFO` | `ut171.ts:345` |

⚠️ Note the **length quirk**: START/DEVICE_INFO carry `len = 0x0004` on the wire, i.e. the length counts `cmd + payload + the 2 checksum bytes` (`1 + 1 + 2 = 4`). The soft-button frames built by `createCmd` (§Controls) instead carry `len = cmd + payload` only. Both START/DEVICE_INFO are reproduced verbatim from the app's captured wire bytes (`ut171.ts:341-345`) rather than rebuilt via `createCmd`, so the on-wire bytes match exactly.

## Frame format

All multi-byte fields are **little-endian**; numeric measurement values are **little-endian IEEE-754 32-bit floats**.

**Container** (`createCmd`, `ut171.ts:36-50`):

```
AB CD <len:u16 LE> <cmd:u8> <payload...> <chk:u16 LE>
```

- `len` = number of body bytes. For the `createCmd`-built soft-button frames `len = 1 + payload.length` (cmd + payload, **excluding** the checksum — `ut171.ts:37`). For the live/stored frames received from the meter the frame total is `4 + len` and `len` counts `cmd + payload + the 2 checksum bytes` (`ut171.ts:12-13`, framer at `ut171.ts:267-268`). (This is the same length quirk noted for START/DEVICE_INFO above; the receive-side framer and the soft-button send-side disagree on whether the checksum is counted in `len`.)
- **Checksum** = `(Σ bytes[2 .. body]) & 0xFFFF`, stored little-endian. It sums the two length bytes + cmd + payload, **excluding** the `0xAB 0xCD` header and the 2 checksum bytes themselves (`ut171.ts:14-15`). Send-side: `createCmd` sums `head[2..head.length-1]` (`ut171.ts:46-49`). Receive-side verification (`checksumOk`, `ut171.ts:306-314`): `Σ(frame[2 .. len-3]) & 0xFFFF == frame[len-2] | (frame[len-1] << 8)`.

**Live measurement frame** (`CMDID=2`, decoded by `decodeUt171`, `ut171.ts:151-252`). Minimum valid length is **17 bytes**: `AB CD len(2) cmd flagA flagB mc ri + main(6) + chk(2)`.

```
offset  bytes  field
 0       1     0xAB                                header
 1       1     0xCD                                header
 2       2     len (u16 LE)                        body length; frame total = 4 + len
 4       1     cmd = 0x02 (CMD_LIVEDATA)           reply cmdID for streamed measurement frames
 5       1     flagA                               status bits (see Decode)
 6       1     flagB                               status bits (see Decode)
 7       1     measureCode                         29 ⇒ OUTPUT/pulse mode; else common-data
 8       1     (range index, unused by decoder)    "ri" — read past; not consumed
 9       4     MAIN value (float32 LE)             primary measured value
13       1     mainFlag                            low nibble = OL state, high nibble = dot/precision
14       1     mainUnitCode                        index into UNIT_TABLE
…              <checksum:u16 LE>                   trailing 2 bytes (Σ as above)
```

The decoder reads the main value at fixed offset `i = 9` (`ut171.ts:209`): `MAIN value(4) @9`, `mainFlag @13`, `mainUnitCode @14` (`ut171.ts:210-212`). The `measureCode` byte is at offset 7 (`ut171.ts:159`).

**OUTPUT/pulse mode** (`measureCode == 29`, `MEASURE_CODE_OUTPUT`, `ut171.ts:104,187`): the frame instead carries a frequency float at offset 9 (`readF32(frame, 9)`, `ut171.ts:188`); duty/width follow but are not decoded into the `Reading` (it is reported as a plain `Hz` reading rounded to 1 decimal, `ut171.ts:192-205`).

**Framing** (`Ut171Framer`, `ut171.ts:258-303`). Buffered AB-CD reassembly mirroring `UT171AManager.checkCmdData`: sync on `AB CD` (`sync`, `ut171.ts:290-302`), read the LE length (`ut171.ts:267`), compute `total = 4 + len` (`ut171.ts:268`). Implausible lengths (`len < 3` or `total > 256`) are treated as a false `AB CD` inside noise — drop one byte and resync (`ut171.ts:269-273`). If the frame is split across notifications (`buf.length < total`) it waits (`ut171.ts:274`). A complete frame whose trailing checksum fails (`!checksumOk`) is treated as a false boundary — shift one byte and resync (`ut171.ts:276-279`). Handles multiple frames per buffer.

**Frame classification** (`classifyKind`, `ut171.ts:316-321`): `cmd == 2` ⇒ `'measurement'`; everything else (incl. `CMDID=114` device-info / record-count replies) ⇒ `'control'`.

**Sniff** (`looksLikeUt171Frame`, `ut171.ts:328-335`): a candidate is a UT171 live frame iff length ≥ 6, header is `AB CD`, `4 + len == bytes.length`, body command is `CMDID=2`, and the trailing LE checksum verifies. This distinguishes it from the UT60BT/UT161 frames on the same ISSC service (those use a big-endian checksum over a fixed 19-byte layout).

## Decode

`decodeUt171(frame, ts)` (`ut171.ts:151-252`) is pure and **never throws**: a frame shorter than 17 bytes, a wrong `AB CD` header, or `cmd != 2` returns a `blank` reading (`ut171.ts:152-155,107-131`). Stored-record frames (`CMDID=3`) and other replies are not measurements, so they also decode to blank.

### Unit table

`mainUnitCode` (offset 14) indexes `UNIT_TABLE` (`getUnitString171`, `ut171.ts:69-102`). Each entry carries a `displayUnit` (with metric prefix), an AC/DC tag, and a range-independent function key. An unknown code falls back to `{ unit: '', acdc: '', fn: '?' }` (`ut171.ts:219`).

| Code | displayUnit | acdc | fn | | Code | displayUnit | acdc | fn |
|---|---|---|---|---|---|---|---|---|
| 0 | `V` | DC | `DCV` | | 16 | `kΩ` | — | `OHM` |
| 1 | `V` | AC | `ACV` | | 17 | `MΩ` | — | `OHM` |
| 2 | `V` | — | `AC+DC` | | 18 | `Hz` | — | `Hz` |
| 3 | `mV` | DC | `DCV` | | 19 | `kHz` | — | `Hz` |
| 4 | `mV` | AC | `ACV` | | 20 | `MHz` | — | `Hz` |
| 5 | `mV` | — | `AC+DC` | | 21 | `%` | — | `%` |
| 6 | `µA` | DC | `DCA` | | 22 | `nF` | — | `CAP` |
| 7 | `µA` | AC | `ACA` | | 23 | `µF` | — | `CAP` |
| 8 | `µA` | — | `AC+DC` | | 24 | `mF` | — | `CAP` |

The display string already embeds the prefix + AC/DC; SI normalization is derived from the display prefix via `unitInfo()` (`ut171.ts:236`, `types.ts:168-174`): `baseValue = displayValue * 10^exp` where `exp` is the prefix exponent (n=−9, µ=−6, m=−3, k=+3, M=+6). E.g. a `16: kΩ` reading of `1.002` → `baseUnit = "Ω"`, `baseValue = 1002`.

### mainFlag (offset 13): OL state + display precision

`mainFlag` splits into a low nibble (overload) and high nibble (decimal places) (`ut171.ts:217-218`):

- `ol = mainFlag & 0x0F`: `0` = normal, `1` = OL (`displayText = "OL"`), `2` = −OL (`displayText = "-OL"`); for OL/−OL `displayValue = null` (`ut171.ts:221-226`).
- `dot = (mainFlag & 0xF0) >> 4`: display rounding precision (decimal places), **not** a divisor. `displayValue = roundDot(mainFloat, dot)` (round to `dot` places, `ut171.ts:141-144,232`) and `displayText = displayValue.toFixed(dot)` (`ut171.ts:233`).

A non-OL but non-finite float (corrupt frame) is treated as overload-ish: `displayText = "OL"`, `displayValue = null` (`ut171.ts:227-230`).

### Flag bits (flagA @5, flagB @6)

| Flag | Bit | Source | Notes |
|---|---|---|---|
| `rel` | flagA `0x10` (bit 4) | `ut171.ts:163` | relative mode active |
| `hold` | flagA `0x80` (bit 7) | `ut171.ts:164` | data hold |
| `lowBattery` | flagA `0x04` (bit 2) | `ut171.ts:165` | |
| `bargraph` | flagA `0x08` (bit 3) | `ut171.ts:184` | bit set ⇒ `bargraph = 1`; ❓ presence flag only — the actual bar count is **not** in the frame |
| MAX/MIN active | flagA `0x20` (bit 5) | `ut171.ts:173` | gates `max`/`min` below |
| PEAK active | flagA `0x40` (bit 6) | `ut171.ts:180` | gates `peakMax`/`peakMin` below |
| `auto` | flagB `0x01` (bit 0) | `ut171.ts:167` | autoranging |
| `hvWarning` | flagB `0x02` (bit 1) | `ut171.ts:168` | high-voltage warning |
| maxMode | flagB `0x60` (bits 5–6) | `ut171.ts:174` | `(flagB & 0x60) >> 5`: `1` = max, `2` = min (`3` = avg, not surfaced) |

**MAX/MIN and PEAK resolution.** `maxMode = (flagB & 0x60) >> 5` selects which of max/min/avg is shown. When the MAX/MIN-active bit (flagA bit 5) is set: `flags.max = (maxMode === 1)`, `flags.min = (maxMode === 2)` (`ut171.ts:175-178`). When the PEAK-active bit (flagA bit 6) is set, the **same** `maxMode` distinguishes peak max/min: `flags.peakMax = (maxMode === 1)`, `flags.peakMin = (maxMode === 2)` (`ut171.ts:180-183`). The `blank` reading defaults all of `max/min/peakMax/peakMin` to `false` (`ut171.ts:122-128`).

### Resulting Reading

`function` = the unit entry's `fn`, `displayUnit` = entry `unit`, `acdc` = entry `acdc` (`ut171.ts:241-248`). `overload` is `true` for OL/−OL. `bargraph` is `0` or `1` (presence only). OUTPUT/pulse frames always report `function: 'Hz'`, `displayUnit/baseUnit: 'Hz'`, `acdc: ''` (`ut171.ts:193-205`); a non-finite OUTPUT frequency yields a blank Hz reading (`ut171.ts:189-191`).

## Verification

**`untested` / `ported-unverified`** (`ut171.ts:377`). Nothing here has been confirmed against a physical UT171 or UT-D07A adapter. Known gaps and inferred mappings to confirm on hardware:

- ❓ **MAX vs MIN bit (`maxMode`)** — `1 = max`, `2 = min` is inferred from the app; the avg case (`maxMode == 3`) and the exact gating against flagA bits 5/6 need confirmation.
- ❓ **PEAK reuses `maxMode`** — the same bits are assumed to distinguish peak max/min; unverified.
- ❓ **Bargraph** — flagA bit 3 is treated as a presence flag only (`bargraph = 0|1`); the real analog bar **count** is not located in the frame.
- ❓ **Frame offsets** — the fixed `i = 9` main-value offset and the offset-8 "range index" byte (read past, not consumed) are inferred from the app's `parseCommonData171`; field widths beyond the decoded ones (duty/width in OUTPUT mode) are not parsed.
- ⚠️ **Length-field inconsistency** — the send-side `createCmd` excludes the checksum from `len`, while START/DEVICE_INFO and the receive-side framer include it. Both are reproduced from captured app bytes; confirm the meter accepts the `createCmd` soft-button frames as-built.
- ⚠️ **Unit codes 27–31** (DIODE/CONT/COND siemens) and the `AC+DC` function entries are ported, not observed.
